# EpicTrace 产品需求

- **日期**:2026-06-09
- **范围**:本文档描述 **产品/功能行为**(端到端流程、各阶段职责、MVP 边界)。技术实现见配套文档 [`2026-06-09-epictrace-tech-stack-architecture-design.md`](./2026-06-09-epictrace-tech-stack-architecture-design.md)。

---

## 1. 产品定位

EpicTrace 是一个**端到端的 AI session memory / knowledge workspace**。它从多模态采集开始,把用户在会议、课堂、讲座、调研、debug、深度工作中产生的声音、截图、剪贴板、笔记和文件,经过暂存、处理、切割、整理归类、入库和索引,最后变成**可对话、可引用、可复用的本地知识库**。

它**不是**单纯的录音软件、笔记软件,也不是普通的文件 RAG 聊天框。它覆盖完整流程:

```text
采集 session → 暂存 raw session → 处理多媒体 → 判断是否需要切割
  → Agent 切割并由用户复核 → Agent 归类到 Project 并组织文件结构
  → 用户复核并提交 → 暂存等待 embedding → 建立知识库索引
  → 通过对话框进行带来源的问答和复用
```

**面向人群**:不只学生,也包括公司员工、知识工作者、工程师、PM、研究人员。

**典型场景**:上课、开会、线上讲座、项目调研、debug、看视频学习、做作业、写代码、一段混合多个工作的深度工作时间。

**核心目标**:把用户主动记录下来的 session 和文件,整理成 **Project 级别的、可检索、可追问、可引用、可复用的 AI 记忆库**。

## 2. 痛点与核心强点

### 2.1 它解决的痛点(现有工具各缺一块)

| 现有做法 | 缺口 |
|---|---|
| **纯采集类工具** | 只做采集 + 规范化整理,**不做处理/不能问答**。资料躺在文件夹里,真要学习还得自己进文件夹用 Claude Code/Codex —— 工具断在"整理好了"那一步。 |
| **直接拿 Claude Code/Codex 进文件夹问** | 小项目还行;**一学期 / 更大规模的资料就崩**:① 烧 token ② 爆上下文窗口。没有索引,每次都把一堆原文硬塞进去。 |
| **文件 RAG 类(如 AnythingLLM)** | 有检索,但**没有采集、没有时间锚定的来源**。答完**跳不回"那一刻"**的讲课录音/截图/剪贴板;也不管你怎么把多模态素材采进来、整理进项目。 |
| **录音软件 / 笔记软件** | 各管一段,**没人打通"采集 → 整理 → 可问答记忆库"的完整链路**。 |
| **多模态上下文散落** | 声音/截图/剪贴板/笔记/文件四处分散,**无法在一条时间线里回溯,也无法作为统一记忆被检索和引用**。 |

### 2.2 核心强点(差异化 / 护城河)

1. **端到端闭环**:采集 → 暂存 → 处理 → 切割 → 归类 → 入库 → 索引 → 带引用问答。**别人只覆盖其中一段,EpicTrace 覆盖全程。**
2. **真正的护城河 = "采集到引用"的闭环**:对你自己采集的多模态 session,做**时间锚定、来源可溯的召回**——提问能**跳回那一刻**的讲课录音 / 截图 / 剪贴板事件 / transcript 时间戳。**这是纯文件 RAG(AnythingLLM 类)给不了的。**(产品叙事应以此开头,而非"我们做 RAG"。)
3. **规模化不烧 token / 不爆上下文**:用 RAG 索引代替"把整个文件夹塞进上下文",**一学期甚至更大的资料也能稳定问答**。
4. **Project = 用户控制的真实本地文件夹**:数据不锁在软件黑盒,硬盘结构贴近人类理解;**Agent 提议、用户确认、系统提交**。
5. **本地优先 + opt-in**:主动开 session 才记录,**不是全天候监控**;原始数据/检索/索引尽量留本地。
6. **流水线可分段、不堵人**:三个暂存区让用户自控节奏,耗电/耗性能步骤由用户触发(对笔记本/移动场景友好)。
7. **时间戳是一等公民**:贯穿时间线展示、切割、回溯、引用跳转、Agentic RAG。

### 2.3 一句话定位(对比)

- **vs 纯采集工具**:它们采集到"整理好就停",EpicTrace 继续走完"可问答、可引用"。
- **vs Notion**:Notion 靠人工组织,EpicTrace 是 Agent 提议 + 多模态采集 + 可问答记忆。
- **vs AnythingLLM**:AnythingLLM 是文件 RAG,EpicTrace 多了**采集闭环 + 时间锚定的来源回跳**。
- **总:采集工具的采集 × Notion 的组织 × AnythingLLM 的 RAG —— 缝合成一个"从采集到引用"的完整本地记忆库。**

## 3. 核心概念:Project

EpicTrace 里只有 **Project** 这一个核心组织概念。

一个 Project = **一个用户创建或指派的本地文件夹** + 一个 title。例如:

- title `CS 2506` → folder `/Users/william/Desktop/CS 2506`
- title `AI Internship Search` → folder `/Users/william/Desktop/AI Internship Search`
- title `React Debugging` → folder `/Users/william/Projects/react-app`

用户自由决定每个 Project 放哪,系统不强制统一根目录。

Agent 的归类工作不是"决定抽象主题",而是建议:内容进哪个已有 Project、是否新建 Project、新 Project 的 title 与位置、Project 内子文件夹结构、文件/截图/笔记/session 片段放哪、生成哪些派生文件(summary / action items / review notes 等)。

**核心原则**:
```text
Project 是用户控制的本地文件夹。
Agent 只提出组织方案。
用户确认后才真正提交。
```

---

## 4. Session 采集

用户**主动**开启 session。开启后可按需开关多种输入源(系统不默认采集一切):

- 麦克风声音
- 系统声音(对应线上 meeting / 线下听课讲座)
- 快捷键截图
- 剪贴板行为
- 用户手动输入的笔记
- 用户拖入的文件
- 后续可扩展的其他上下文输入

**所有输入源产生的操作和事件都必须带时间戳。** 时间戳用于:时间线展示、session 切割、原始上下文回溯、来源引用、RAG 检索、对话回答跳转到原始位置。

---

## 5. 采集后暂存区

session 结束后**不立刻强迫整理入库**,先进入**采集后暂存区**(保存还没处理的 raw session)。用户可选:现在处理 / 暂时不处理 / 直接开新 session。

**理由**:整理数据费事;不堵用户;产品像流水线(先采集后处理);多媒体处理耗性能;何时处理交给用户;对移动设备/续航友好。

**职责**:
1. 展示 raw session
2. 以**图形化时间线**查看 session 内容
3. 处理多媒体内容
4. 判断这个 session 是否涉及多个 Project
5. 涉及多个 → 进入切割流程;不涉及 → 直接进入整理归类流程

时间线需能展示关键事件:transcript 片段、截图、剪贴板事件、拖入文件、用户笔记、音频段落、其他带时间戳事件。用户需能通过时间线理解 session 结构(否则无法判断是否切割,也无法给 Agent 有效描述)。

---

## 6. 多媒体处理

若 session 含需处理的多媒体(图片、PDF、PPT 等),先做 OCR / 解析 / 其他处理。

- **有未处理多媒体 → 阻塞下一步提交**,提示用户先处理。
- 无多媒体,或全部处理完 → 不阻塞。
- 多媒体处理**由用户决定何时开始**,不在 session 一结束就强制。

---

## 7. 判断是否涉及多个 Project

暂存区询问用户:「这个 session 是否涉及多个 Project?」

- **否** → 整个 session 作为整体进入整理归类。
- **是** → 进入切割流程。
- **是,并提供描述** → 告诉 Agent 大概怎么切(如:"先 CS2506 复习,然后 AI 实习项目构思,最后 React bug 调试")。

若选"否"或未提供任何切割描述 → 默认整段进入整理归类。

---

## 8. 切割流程

切割 Agent 依据:session 时间线、transcript、截图、OCR、笔记、剪贴板事件、拖入文件、用户描述,提出切割方案。

方案需说明:切成哪些段、每段起止时间、每段大概对应哪个 Project/工作、每段关联哪些截图/笔记/剪贴板/文件/transcript 片段。

### 切割结果复核(强制)

复核界面把切割点显示在**图形化时间线**上,切割点是**可点击的按钮/标记**:
- 点击某切割点 → 时间线自动跳转到对应时间。
- 用户可滚轮查看该时间点附近上下文。

用户可:接受方案 / 否决某切割点 / 手动修改切割点 / 打回 Agent 重做 / 打回时附加新提示词 / 继续下一步。

不需要切割则跳过此流程。

---

## 9. 整理归类前暂存区

切割完成后(或无需切割时)进入**整理归类前暂存区**,确认是否继续归类。

进入整理归类 Agent 前,提供一个**可选提示词输入框**:
- 直接 Enter 留空 → Agent 自动整理。
- 输入提示词 → 告诉 Agent 怎么归类。
- 暂时不处理 → 放回暂存区。

提示词**都非必填**。体验应像 Codex / Claude Code 的 GUI:一路留空按 Enter 也能继续,只在想加约束时才输入。

---

## 10. 整理归类 Agent

职责:把 session(或切割后的段落)组织进 Project。方案包括:
- 进哪个已有 Project / 是否新建 Project / 新 Project 的 title
- Project 文件夹由用户选择或确认
- Project 内如何建子文件夹
- 哪些文件/截图/笔记/transcript 片段放哪
- 是否生成 summary / action items / review notes / quiz / 项目上下文包等派生文件
- 哪些内容进入后续 embedding / RAG 索引

Agent 可自动提出 Project 内文件与目录组织方案,但**所有方案需用户复核**。

### 归类结果复核

用户可:接受全部 / 对某些条目打叉 / 修改某些归类 / 修改 Project / 修改子文件夹位置 / 打回重做 / 打回时附加 prompt。确认后才完成入库。

原则不变:**Agent 提议,用户确认,系统提交。**

---

## 11. 完成入库

用户确认后内容正式入库,按整理后的 Project 文件夹结构呈现在**本地文件系统**里。

**硬盘实际存储应尽量贴近用户看到/理解的 Project 结构**,而非只存在软件黑盒里。用户能在文件系统中看到自己的 Project 文件夹、子文件夹和文件。

Session-based view 仍存在,主要由软件基于记录和元数据在界面里还原。这样同时满足:Project 文件夹对人类清晰、session 叙事在软件里可回溯、session 片段可引用到对应 Project、Project 中文件可引用回原始 session 时间点。

---

## 12. Embedding 暂存区

完成入库后**不一定马上建 embedding**,可进入 **embedding 暂存区**。用户可选:现在建索引 / 稍后 / 批量处理多个入库内容 / 等设备空闲或插电时再处理。

**理由**:embedding 耗性能、耗电,移动设备需考虑续航;用户可能只想先整理、之后统一索引。**入库与建索引是两个可分离的步骤。**

---

## 13. 文件直接入库

除 session 入库外,支持**文件直接入库**(PDF / PPT / 图片 / Markdown / 代码 / 文档 / 其他资料)。不走 session 采集和切割,直接进整理归类:

```text
提交文件 → 整理归类 → 归类结果复核 → 完成入库 → Embedding 暂存区 → 用户确认后建索引
```

用户可给文件附自然语言描述(如:"这是 5 月 13 日 CS2506 关于 virtual memory 的 PPT")。Agent 据描述和文件内容提归类方案。

文件入库需记录**入库时间和入库方式**(session 入库 / 直接文件入库 / 拖入文件)。具体 metadata 见技术文档。

---

## 14. 对话与复用

建立知识库后提供对话入口(可像 AnythingLLM 或 ChatGPT 网页端)。用户选一个或多个 Project 提问。

能力:
- 基于 Project 的知识库问答
- 基于 session 的回溯问答
- 回答**带来源引用**
- 引用能跳回原始 session 时间点 / 截图 / 文件 / 笔记
- 总结 / 行动项生成 / 复习材料生成 / quiz 生成 / 项目上下文包生成
- 把整理后的上下文交给其他 AI agent 使用

**对话页附件**(类 ChatGPT/Claude 的文件/图片提交):支持。需区分**本次对话临时附件**(用完即弃) vs **入库进 Project**(走归类流程)。

对话部分不是简单聊天框,而是连接 Project 文件夹、session 时间线、RAG 索引和来源引用的统一入口。

---

## 15. Agentic RAG

提问后系统**不只做一次 top-k 检索再让 LLM 回答**。Agent 动态决定如何查找和扩展上下文,支持:
- 语义搜索 / 关键词搜索 / 按时间窗 / 按 session / 按 Project
- 找到相关片段后扩展前后文
- 回到原始 session 时间点
- 引用截图、笔记、PDF、PPT、transcript 时间戳
- 判断当前证据是否足够;不足则改写 query 或扩展上下文
- 限制检索轮数和上下文预算
- 找不到时明确说找不到,**不编造**

最终回答尽量带来源引用。(实现:LangGraph 的 ReAct + Reflection 循环 + Rerank,见技术文档。)

---

## 16. 隐私与本地优先

涉及录音、系统声音、截图、剪贴板和本地文件,**隐私是核心要求**。

- 产品必须 **opt-in、session-based**:用户主动开 session 才记录;**不是全天候监控**。
- 默认尽量**本地保存和本地处理**:原始音频、transcript、截图、OCR、剪贴板记录、笔记、文件、本地索引、embedding。
- LLM 可用云端模型(端侧 LLM 质量可能不够)。
- 云端 LLM **不应默认看到完整 session 或完整知识库**。

理想方式:
```text
原始数据留本地。检索和 embedding 尽量本地完成。
只有用户主动提问或执行 Agent 操作时,才把少量相关上下文发送给云端 LLM。
```

> 实现现实(见技术文档):对话/Agentic LLM 采用云端(DeepSeek)换质量,故硬隐私承诺基本放弃;但检索/embedding 本地完成,且只把检索命中的少量片段上云。`LLMProvider` 抽象保留"以后接本地 LLM"的口子。

---

## 17. 文件移动和修改同步(不进 MVP)

文件移动、重命名、修改后的自动同步**不放进 MVP**。MVP 不做:
- 实时监听文件系统变化
- 自动识别手动移动 / 改名
- 自动处理文件内容变更后的索引同步
- 类 Git 的版本追踪系统

但**入库和引用所需的基础记录必须有**。后续若发现文件缺失或内容变化,MVP 先做提示或状态标记,不做复杂自动修复。

---

## 18. MVP 边界

### MVP 包括
- 创建 / 结束 session;session 输入源开关;记录带时间戳的事件
- 采集后暂存区;raw session 图形化时间线
- 多媒体处理状态和阻塞提示
- 判断是否涉及多个 Project;可选切割描述
- Agent 切割 session;切割结果复核;手动修改/打回切割方案
- 整理归类前暂存区;可选整理归类提示词
- 整理归类 Agent;归类结果复核
- Project 创建/选择/指派本地文件夹;Project 内部子文件夹/文件组织建议
- 完成入库
- Embedding 暂存区;用户确认后建立 embedding / RAG 索引
- 文件直接入库;文件入库方式和时间记录
- 基础带引用问答;基于 Project 的对话框

### MVP 暂不包括
- 文件移动/修改的自动同步;类 Git 的版本追踪
- 团队协作;云同步
- 全自动无确认文件移动
- 大而全的 Notion 编辑器
- 永远开启的后台监控
- AIGC 检测(与产品不契合,见技术文档 §11)

---

## 19. 一句话总结

EpicTrace 是一个端到端的 AI session memory workspace。用户主动开启 session 后,可按需采集声音、截图、剪贴板、笔记和文件;session 结束后进入暂存区,经过多媒体处理、可选切割、Agent 归类、用户复核和入库,内容被整理进用户指定的本地 Project 文件夹。之后用户可选择建立 embedding / RAG 索引,并通过类似 ChatGPT 或 AnythingLLM 的对话框,对一个或多个 Project 进行带来源引用的问答、总结和复用。
